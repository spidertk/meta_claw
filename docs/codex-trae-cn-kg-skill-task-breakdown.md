# Codex / Trae CN 知识图谱技能阶段任务清单

## 1. 文档目标

本文将 [codex-trae-cn-kg-skill-prd.md](/Users/kai/IdeaProjects/meta_claw/docs/codex-trae-cn-kg-skill-prd.md) 拆解为可执行的阶段任务清单。

目标是让后续实现按阶段推进时，团队能明确：

- 当前阶段要完成什么
- 哪些任务是前置依赖
- 哪些任务可以延后
- 什么状态算阶段完成

## 2. 使用方式

每个阶段按四类任务组织：

- 产品任务：明确用户体验和外部行为
- 技术任务：实现底层能力
- 集成任务：打通平台、目录结构或上下游模块
- 验收任务：确认阶段真正可交付

状态建议使用：

- `TODO`
- `DOING`
- `DONE`
- `BLOCKED`

- `TODO` 明确默认命令形态

- `TODO` 明确首版用户只需要一个主命令，不暴露复杂模式

### 3.3 技术任务

- `TODO` 复用 `.rwa/graphify` 的检测、抽取、建图、报告、导出流程
- `TODO` 封装一个面向新技能的统一调用入口
- `TODO` 约定输出目录为 `graphify-out/`

- `TODO` 处理 graphify 不存在或依赖缺失时的安装 / 提示逻辑

### 3.4 集成任务

- `TODO` 生成 Codex 技能文件
- `TODO` 生成 Trae CN 技能文件

### 3.5 验收任务

- `TODO` 用普通目录验证可生成完整 `graphify-out/`
- `TODO` 用代码仓库验证可生成结构图谱

## 10. 建议实施顺序

建议实际执行顺序如下：

1. 完成 Phase 1，拿到最小可用入口
2. 完成 Phase 2，确保能进入日常工作流
3. 完成 Phase 3，让图谱开始进入知识库生命周期
4. 完成 Phase 4，让图谱开始驱动知识动作
5. 完成 Phase 5，让图谱接入 `llm_wiki` 工作流
6. 最后做 Phase 6 的平台增强和高级能力

## 12. 结论

这份任务清单的核心原则是：

- 不删功能
- 不一次做完
- 每个阶段都形成用户可感知价值
- 后续实现严格按阶段推进

这样做的好处是，既保留了完整产品愿景，也避免了在第一轮实现时被复杂度拖垮。
