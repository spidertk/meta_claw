# 私有知识中台设计文档

日期：2026-04-20

## 1. 文档目的

本文定义 `meta_claw` 第一阶段“私有知识中台”设计方案。

目标不是直接实现完整私有智能体，也不是重写 `graphify` 或 `llm-wiki`，而是基于现有分析结果，设计一个可持续积累、可增量维护、可审计、可扩展的知识底座，用于：

- 通过 `.rwa` 持续录入文件、代码仓库和受控外链副本
- 用结构化图谱维护领域知识的深度、广度和变化
- 用 Wiki 将高价值知识沉淀为人类可读资产
- 优先服务 `Codex` 与 `Trae` 这类 CLI coding agent
- 为后续私有模型接入预留稳定扩展点

## 2. 设计来源与继承思想

本次设计明确继承三类已有思想。

### 2.1 `.rwa/llm-wiki-md/graphify-out`

该分析结果提供了基础方法论：知识系统应由三层组成。

- `Raw Sources`：原始资料层
- `Wiki Layer`：持续沉淀的人类可读知识层
- `Schema / Ingest / Query / Lint`：维护操作层

继承原则：

- 知识不能停留在原始资料，需要沉淀成持续维护的 wiki
- 系统必须围绕操作闭环设计，而不是一次性导入
- wiki 是长期资产，不是单次问答副产物

### 2.2 `.rwa/llm-wiki-skill/graphify-out`

该分析结果提供了执行框架：一个可运转的知识维护系统应围绕五类持续操作组织。

- `compile`
- `ingest`
- `query`
- `lint`
- `audit`

继承原则：

- Obsidian 作为唯一的人类阅读和反馈入口已经足够
- 人的修正意见必须以可追溯文件形式进入系统长期记忆
- wiki 需要按主题、实体、摘要分层组织，避免堆积式大文档

### 2.3 `.rwa/graphify/graphify-out`

该分析结果提供了结构化知识底座能力。

- 检测、抽取、建图、聚类、diff、report
- 社区、god nodes、bridge nodes、弱连接点分析
- update、cache、watch/hook、增量维护思路

继承原则：

- `graph.json` 是结构化知识底座，优先给模型和程序消费
- 领域深度和广度应由图谱分析信号驱动
- 增量维护必须围绕快照、diff 和受影响范围传播设计

## 3. 产品目标与接入优先级

### 3.1 第一阶段目标

第一阶段要实现一个“私有知识中台”最小可用闭环，使系统能够：

- 通过 `.rwa` 受控录入本地文件、本地仓库和受控外链副本
- 基于 `graphify` 维护结构化图谱和增量变化
- 基于 `llm-wiki` 思想沉淀面向人的 wiki
- 通过 Obsidian 提供唯一的人类阅读、导航和审校入口
- 在知识持续增长时，自动发现待更新领域和待维护页面

### 3.2 长期目标

长期目标不是仅维护“项目文档”，而是为私有智能体建立长期知识底座。

未来输入源可扩展为：

- 代码仓库
- 本地文件和文档
- 网页和 PDF
- 视频链接，例如抖音
- 临时感兴趣的零散知识来源

长期价值目标是：

- 持续积累私有领域知识，而不是每次重新检索
- 让私有智能体在相关领域问题上，比通用智能体更了解用户的长期积累
- 在有限上下文里，通过更好的知识组织和主题归档，让模型看见更全面的相关私有知识
- 提升理解问题、回答问题和解决问题的质量，并降低长期开发和使用成本

### 3.3 接入优先级

为控制成本并优先服务高频工作流，本项目的消费侧接入优先级为：

1. `Codex`
2. `Trae`
3. 私有模型独立接入

其中：

- `Codex` 和 `Trae` 是一期优先服务对象
- 私有模型接入不在本期实现范围内
- 但底层数据、状态和输出格式必须从一期开始预留私有模型扩展点

## 4. 目标边界

### 4.1 In Scope

第一阶段明确包含：

- `.rwa` 作为唯一受控录入层
- 本地文件、本地目录、本地 git 仓库录入
- 受控外链录入，默认落地为 `.rwa` 内本地副本
- 仅在必要时使用指针文件表示未完整纳入的大文件或敏感内容
- 基于“来源身份 + 内容指纹”的混合策略做增量判定
- 用 `graphify` 维护结构化知识图、差异信息和领域分析信号
- 用 wiki 维护面向人的知识沉淀、索引、日志和审校面
- Obsidian 作为唯一人类入口
- 半自动触发：录入手动触发，待更新内容和受影响领域自动发现

### 4.2 Out of Scope

第一阶段明确不包含：

- 独立 HTML wiki/viewer
- 全自动持续抓取全网
- 完整的模型任务上下文打包能力
- 私有模型调用、路由、鉴权和部署
- 复杂多用户协作和权限系统

### 4.3 非目标

本阶段不追求：

- 一次性覆盖所有知识源类型
- 完整复刻 `llm-wiki` 桌面交互能力
- 直接把大图谱展示给人阅读
- 让系统对每次变化都做全量重建

## 5. 总体方案

### 5.1 方案选择

本次采用“图谱底座 + wiki 投影 + Obsidian 人类入口”的双层协同方案。

- `.rwa` 负责受控录入
- `graphify` 负责结构化知识底座和增量判断
- `wiki` 负责面向人的长期沉淀
- `Obsidian` 负责浏览、导航和人工审校

该方案不把图谱直接当人类界面，也不让 wiki 承担全部结构判断，而是让两者各司其职。

### 5.2 核心原则

- `受控录入优先于临时抓取`
  只有进入 `.rwa` 的内容才进入长期知识系统。

- `图谱是底座，wiki 是投影`
  图谱保存完整结构，wiki 只呈现经过组织后的稳定知识。

- `增量维护优先于全量重建`
  先识别变化，再更新受影响子图和相关 wiki 页面。

- `领域深度来自持续迭代`
  每次录入、更新、lint、audit 都只推进一部分，但系统持续发现薄弱领域、桥接点和待补主题。

- `抽取器可编排而非写死单实现`
  抽取层必须允许多个实现串联、替换或并存，不能把框架绑定到单一抽取器。

- `超大来源分层处理，展示层分层投影`
  对 10W+ 代码仓库、超大 PDF/PPT、长视频等来源，系统先建立骨架和分段导航，再按需深挖，禁止把全量结果直接堆到人类展示层。

## 6. 系统架构

系统分为五层，每层只负责一件事。

### 6.1 Source Registry

录入层，唯一入口是 `.rwa`。

职责：

- 接收文件、目录、仓库、网页/PDF 链接等来源
- 生成标准化来源记录
- 管理来源元数据和录入状态

### 6.2 Identity + Snapshot

来源身份与快照层。

职责：

- 为每个知识源维护稳定 `source_id`
- 为每次同步生成 `content_fingerprint`
- 保存来源历史快照和结构指纹

### 6.3 Graph Extraction

结构化抽取层，由 `graphify` 能力优先承担，但架构上不能把 `graphify` 写死为唯一实现。

职责：

- 检测来源类型
- 对代码走 AST / 静态结构优先抽取
- 对文档走文本与语义结构抽取
- 维护图谱、社区、差异和领域分析信号

#### 6.3.1 可编排抽取层

抽取层必须设计成一条可替换、可组合、可分层降级的抽取流水线，而不是单个工具调用。

建议分为三层：

- `ingest-normalize`
  将来源转成统一输入单元，例如仓库文件树、文档 chunk、视频转写块。

- `extractor-pipeline`
  允许多个抽取器串联或分支运行，例如检测器、结构抽取器、语义抽取器和知识增强器。

- `knowledge-merge`
  将多个抽取结果合并成统一知识状态，再进入 graph / wiki。

#### 6.3.2 抽取器接口契约

每个抽取器必须遵守统一契约，不直接写最终 wiki，也不直接耦合其他具体实现。

输入至少包括：

- `source_ref`
- `snapshot_ref`
- `unit_ref`
- `processing_scope`
- `budget_hint`

输出至少包括：

- `extracted_units`
- `relations`
- `summaries`
- `artifacts`
- `coverage`
- `confidence`
- `issues`

后续新增或替换抽取器时，只要仍遵守这套输入输出契约，就必须能够接入同一框架。

#### 6.3.3 超大项目与超大文件策略

对超大来源，系统默认采用“全局骨架优先，局部深挖按需触发”的策略，而不是一次吃完整来源。

统一分三层处理：

- `L1 结构层`
  只抽全局骨架。
  对仓库是目录树、模块、入口。
  对 PDF / PPT 是章节、页组、目录、主题段。
  对视频是时间段、章节段、转写块索引。

- `L2 摘要 / 热点层`
  抽每个结构块的摘要、关键词、关联主题和热点单元。

- `L3 深度内容层`
  只对用户关注区域、变化区域或高价值区域做深抽。

#### 6.3.4 超大来源切块与展示约束

任何超长来源都必须先切块，再决定是否深挖。切块后的每个单元都应有独立 `unit_ref`，方便增量更新、局部重抽、局部展示和局部进入 wiki。

展示层必须遵守以下约束：

- 人类展示层禁止直接渲染完整大图或完整超长内容
- Obsidian 侧只展示分层导航结果，不展示全量原始聚合
- wiki 页面必须遵守“索引页 + 子页”结构
- 超大来源必须优先展示总览页、章节 / 分段页、热点页和变更页
- 只有用户继续下钻时，才展示更细内容

### 6.4 Wiki Synthesis

知识沉淀层。

职责：

- 将图谱变化转成 wiki 更新任务
- 维护 summary、concept、entity、project、topic 页面
- 更新索引、日志和审校面
- 控制页面粒度，避免大页失控

### 6.5 Obsidian Workspace

人类消费与反馈层。

职责：

- 浏览 wiki
- 基于目录、双链和局部关系理解知识
- 发起修订、补充和审校

## 7. 关键模块与数据流

第一阶段系统收敛为 6 个关键模块。

### 7.1 `ingest_registry`

输入：

- 本地文件
- 本地目录
- 本地 git 仓库
- 网页/PDF 链接

输出标准化来源记录，至少包括：

- 来源类型
- 原始地址
- 本地落地路径
- 稳定 `source_id`
- 最近一次抓取时间
- 当前状态

### 7.2 `snapshot_manager`

职责：

- 将来源变成可比较快照
- 计算内容指纹
- 比较新旧快照

快照粒度：

- 单文件：文件本身
- 目录：目录树 + 文件摘要
- 仓库：工作副本 + commit + 关键文件指纹
- 网页/PDF：下载副本 + 提取文本 + 元数据

### 7.3 `graph_update_engine`

职责：

- 根据变化来源集合执行局部图谱更新
- 产出新增/删除/变化节点边
- 识别受影响社区、桥接点、薄弱领域

### 7.4 `wiki_planner`

职责：

- 将结构变化转成知识维护动作
- 生成待处理任务，而不是直接写 wiki

典型任务包括：

- 为新来源生成 summary 页
- 为扩大领域拆分子主题页
- 为跨领域新连接补交叉主题页
- 为失效或大改来源标记待重审页面

### 7.5 `wiki_synthesizer`

职责：

- 真正执行 wiki 写入和更新
- 维护 `index.md`、专题页、实体页、日志和审校记录

### 7.6 `review_audit_loop`

职责：

- 在 Obsidian 中承接人工阅读和纠偏
- 将反馈回流到下一轮知识维护链路

### 7.7 主数据流

主链路如下：

`.rwa source`
-> `source registry`
-> `snapshot manager`
-> `changed source set`
-> `graph update engine`
-> `graph diff + domain signals`
-> `wiki planner`
-> `wiki synthesizer`
-> `Obsidian review/audit`
-> `next iteration`

## 8. 增量维护机制

系统必须自行维护一套独立于原始来源的状态库，用于识别来源身份、比较版本快照、定位内部变化单元，并把变化传播到知识图谱和 wiki。增量判断基线由 `meta_claw` 保存，不依赖外部来源自行提供知识级变更信息。

### 8.1 核心标识

每个来源至少维护两类标识：

- `source_id`
  来源稳定身份。对路径、仓库、URL 等来源生成稳定标识，用于判断“这是不是同一个知识源”。

- `content_fingerprint`
  当前内容快照指纹，用于判断“同一来源是否发生内容变化”。

### 8.2 三层状态

系统至少维护三类状态。

#### 8.2.1 `source_registry`

记录来源类型、原始地址、本地路径、首次录入时间、最近同步时间和活动状态。

#### 8.2.2 `snapshot_store`

记录版本快照及其结构信息。

对仓库来源，至少记录：

- 仓库地址
- 分支
- commit SHA
- 文件树摘要
- 文件级内容指纹

对文档来源，至少记录：

- 原始文件 hash
- 抽取文本 hash
- 章节树或内容块指纹

对网页来源，至少记录：

- 抓取时间
- 原始 URL
- 正文内容 hash
- 正文块级指纹

#### 8.2.3 `knowledge_state`

记录从快照抽取出的知识结果及其依赖关系，包括：

- 图谱节点和边
- 社区归属
- 主题覆盖状态
- wiki 页面依赖映射
- 上次分析深度和更新时间

### 8.3 四层 diff

系统的增量比较必须分四层执行。

#### 8.3.1 `source-level diff`

判断当前输入是新来源，还是已存在来源的新同步版本。

#### 8.3.2 `snapshot-level diff`

比较新旧快照，判断是无变化、小变化还是大版本变化。

#### 8.3.3 `unit-level diff`

定位来源内部变化的最小单元。

- 仓库：文件、目录、符号、调用关系集合
- PDF / 文档：章节、段落块、文本块
- 网页：正文结构块、语义段落

#### 8.3.4 `knowledge-level impact`

将内部变化传播到知识层，识别受影响的：

- 节点和边
- 社区
- 主题页
- 摘要页
- 领域页

### 8.4 仓库来源的增量规则

仓库来源采用 `repo -> commit -> file/symbol` 增量链路。

- 仓库地址定义稳定 `source_id`
- commit SHA 作为版本快照主键
- 通过新旧 commit 对比识别新增、修改、删除、重命名文件
- 仅对变化文件及相关符号重新抽取
- 将变化传播到相关概念、模块、领域和 wiki 页面
- 删除或重命名不直接导致 wiki 物理删除，而是先标记为失效、待重组或待迁移

### 8.5 文档来源的增量规则

PDF / 长文档采用 `document -> snapshot -> section/chunk` 增量链路。

- 来源身份由 URL、标题、登记记录等元数据共同确定
- 文件 hash 用于识别文档版本变化
- 章节目录、页数、文本块 hash、术语分布等指标用于识别变化范围
- 小范围变化只重算受影响章节及其关联知识
- 结构和内容大范围变化时，判定为同一来源的大版本更新
- 大版本更新触发摘要页重写、相关概念页重审和版本差异总结生成

### 8.6 增量结果类型

系统必须支持以下增量结果类型：

- `unchanged`
- `partial_update`
- `major_update`
- `new_source`

### 8.7 增量目标

增量维护的目标不是单纯减少计算，而是保证“知识更新最小化且有依据”：

- 只重跑必要抽取
- 只重写受影响 wiki 页面
- 对重大变化保留版本语义
- 自动生成知识缺口和待处理任务

## 9. Wiki 组织与人类工作流

### 9.1 Wiki 定位

wiki 是给人看的长期知识资产，不承担全部机器结构表达责任。

### 9.2 页面类型

第一阶段建议至少维护：

- 领域首页
- 概念页
- 实体页
- 项目/模块页
- 来源总结页
- 变更日志
- 审校记录

### 9.3 页面组织原则

- 领域页负责导航，不承载全部细节
- 子页承载具体知识
- 图谱社区不能原样映射为大页
- 页面之间通过 wikilink 建立稳定关系

### 9.4 人类入口

Obsidian 是唯一人类入口。

人类通过 Obsidian：

- 浏览 wiki
- 按领域导航
- 查看局部连接
- 发起修订和审校

系统不以完整大图可视化作为人类主界面。

## 10. 错误处理与恢复策略

系统必须把错误分为四类处理，避免单点失败污染整条链路。

### 10.1 来源错误

包括路径失效、仓库无法同步、下载失败、文件损坏、权限不足等。

处理原则：

- 保留来源记录
- 标记本次同步失败
- 不覆盖上一次有效快照
- 不触发后续知识删除
- 记录失败原因、时间和重试建议

### 10.2 抽取错误

包括类型识别失败、文本抽取失败、解析器异常、建图失败等。

处理原则：

- 失败隔离到来源或最小处理单元
- 允许批次部分成功
- 标记当前来源抽取不完整
- 不把未完成状态误报为已全面更新

### 10.3 映射错误

包括来源身份冲突、内容指纹异常漂移、重命名误判、知识单元关联失败、wiki 依赖映射缺失等。

处理原则：

- 可疑情况降级为待人工确认
- 不做高风险自动合并
- 将冲突转成待处理任务或审校项

### 10.4 沉淀错误

包括页面写坏、页面重复、索引失配、链接失效、领域页过大、删除传播过度等。

处理原则：

- wiki 更新可审计
- wiki 更新可回滚
- wiki 更新可重新生成

### 10.5 保护机制

第一阶段至少具备以下保护机制：

- `snapshot_immutability`
  已成功快照不能被失败同步覆盖。

- `partial_batch_tolerance`
  一批来源中部分失败时，其余成功来源照常进入后续流程。

- `rebuild_escape_hatch`
  当增量状态异常时，允许对单个来源、单个领域或全局执行受控重建。

## 11. 测试策略

测试按层拆开，不只做 end-to-end 冒烟。

### 11.1 `source registry / snapshot` 测试

验证：

- 来源登记
- 重复录入
- 路径规范化
- 指纹计算
- 快照比较

重点覆盖：

- 同一路径重复录入不会生成新来源
- 同一 URL 新版本会生成新快照
- 内容未变时不误报更新
- 指针来源不被误当作完整副本

### 11.2 `incremental diff` 测试

验证四层 diff 是否稳定。

重点覆盖：

- 仓库 commit 变化能正确识别新增、修改、删除、重命名文件
- PDF 小改动只影响局部章节
- PDF 大改版触发 `major_update`
- 相同内容换路径不会被误判为全新来源

### 11.3 `graph update` 测试

验证局部变化只影响局部图谱。

重点覆盖：

- 新增文件只增加相关节点和边
- 删除文件不会立刻造成不受控知识塌缩
- 社区、桥接点和薄弱领域信号在局部更新后仍可计算
- 图谱状态可重复生成

### 11.4 `wiki planner / synthesizer` 测试

验证知识变化能正确转成 wiki 动作。

重点覆盖：

- 新来源生成 summary 页和导航入口
- 受影响主题页被标记待更新
- 过大的领域页触发拆分建议
- 删除来源时页面先降级或标记，而不是直接硬删

### 11.5 `lint / audit` 测试

验证人类可读层健康度。

重点覆盖：

- 坏链接、孤立页、索引缺失可被发现
- audit 能关联到目标页面
- 页面更新日志可追溯来源变化
- Obsidian 侧结构稳定可导航

### 11.6 `end-to-end` 测试

第一阶段至少需要两类样本：

- 一个 git 仓库样本
- 一个 PDF / 长文档样本

### 11.7 验收标准

本节验收标准：

- 任一来源同步失败时，不破坏已有可用知识状态
- 任一局部变化只触发受影响范围内更新
- 任一 wiki 更新都能追溯其来源快照和图谱变化依据
- 任一异常状态都能通过重试、降级或局部重建恢复

## 12. 知识失真风险与防错原则

本系统在超大来源切块、局部增量更新、聚合摘要和分层展示过程中，存在“局部合理但整体失真”的天然风险。为避免知识体系不完整、上下文误判和高层结论错误升级，本设计增加独立的防错原则章节，作为后续实现和计划拆分时的硬约束。

### 12.1 禁止局部冒充整体

系统必须明确区分以下层级：

- 局部观察
- 局部摘要
- 主题级结论
- 领域级结论

任何来自单个 `unit_ref` 或少量局部块的结果，都只能先作为候选知识，不能直接升级成主题页或领域页结论。只有经过父层聚合和一致性校验后，才能进入较高层级知识资产。

### 12.2 禁止无覆盖声明的结论

凡是基于切块、热点区域、局部章节、局部模块得到的结论，都必须带：

- `coverage`
- `scope`

如果系统无法证明当前覆盖了整个来源或整个主题，就不能输出看起来像完整结论的表述。系统可以说明“当前只覆盖认证模块”或“当前仅覆盖第 3 章到第 5 章”，但不能将其伪装成整体事实。

### 12.3 禁止展示层全量堆叠

对超大项目、超大 PDF / PPT、长视频等来源，展示层只能呈现：

- 总览
- 分层导航
- 热点
- 变更
- 下钻入口

系统禁止直接将以下内容整体堆叠到 Obsidian 页面中：

- 全量关系图
- 全量 chunk 摘要
- 全量章节拼接内容
- 全量视频转写结果

原因是展示层一旦失控，人类无法有效校验，系统也更容易将杂乱聚合误当结构化知识。

### 12.4 禁止抽取器私有语义泄漏到框架层

抽取器可以替换、串联、并存，但框架层不能依赖某个抽取器的私有字段、私有推理方式或内部状态。否则以后更换实现时，数据仍在，语义解释却会断裂。

因此框架层只能依赖：

- 统一输入输出契约
- 统一 `unit_ref`
- 统一 `coverage / scope`
- 统一聚合与校验规则

### 12.5 禁止增量更新直接破坏旧结论

当来源发生局部变化时，系统不能因为只重跑了局部块，就直接推翻整个主题结论。

正确流程应为：

1. 先标记受影响范围
2. 将旧结论降级为“待确认”或“部分过期”
3. 在父层重新聚合
4. 最终决定保留、修订或拆分旧结论

这样可以避免部分变化导致整体知识突然漂移。

### 12.6 核心防错原则

后续实现必须遵守以下五条原则：

- `结构边界优先于长度边界`
- `局部结论不能直接升级为整体结论`
- `所有高层知识都必须可追溯到聚合过程`
- `展示层只展示人能验证的分层投影`
- `增量更新优先标记影响，再重算结论，不直接覆盖`

## 13. 长期定位与扩展点

### 12.1 长期定位

本系统第一阶段按“知识中台”设计边界，但数据模型、状态管理和输出接口必须天然支持后续作为私有智能体外挂知识底座使用。

短期目标：

- 围绕 `.rwa`、`graphify`、wiki 和 Obsidian 建立可持续维护的知识闭环

长期目标：

- 持续吸收代码、文档、网页、PDF、视频和临时兴趣来源
- 将这些私有知识沉淀为可追踪、可增量更新、可被主题归档和可被智能体消费的长期资产

### 12.2 必须预留的扩展点

#### 12.2.1 `source_adapters`

录入层必须允许未来增加：

- 视频链接适配器
- 音频来源适配器
- 聊天记录来源适配器
- 收藏夹、剪藏和零散条目适配器

视频类来源后续应支持：

`链接 -> 下载/转录/摘要 -> 结构化 source`

#### 12.2.2 `multimodal_snapshots`

快照层不能只假设输入是文本文件。状态模型必须允许一个来源包含：

- 原始媒体
- 转写文本
- 摘要文本
- 提取元数据
- 派生知识结果

#### 12.2.3 `topic_routing`

新增来源后续应能自动：

- 归入已有主题知识库
- 或识别需要创建新主题

第一阶段需要把主题归档规则和知识页依赖映射做成独立层。

#### 12.2.4 `agent_facing_outputs`

第一阶段不做完整私有 agent 集成，但应保持稳定中间表示，便于后续提供：

- 领域摘要
- 主题候选上下文
- 来源追溯
- 变更提示
- 个人长期兴趣画像

### 12.3 私有模型接入约束

私有模型接入不属于本次实现目标，但必须预留稳定扩展点。

本期要求：

- 不实现私有模型调用、鉴权、路由和部署
- 不将知识输出绑定到 `Codex` / `Trae` 私有格式
- 保持状态层、图谱层、wiki 层和后续上下文输出层具有稳定中间表示

二期接入应能在不重做底座的前提下补上：

- 私有模型适配器
- 面向模型的上下文裁剪/打包
- 领域知识检索与主题路由
- 模型回答后的反馈回流链路

## 14. Phase 路线图

当前设计的 TODO 必须按阶段组织，而不能只保留一组平铺清单。

本设计确定以下阶段边界：

- 一期实现范围 = `Phase 1 + Phase 2`
- 一期扩展预留 = `Phase 3 + Phase 4`
- 后续 implementation plan 只从 `Phase 1 + Phase 2` 的 TODO 中生成

### 13.1 Phase 1：知识源闭环

目标是先把“来源进系统”这件事做稳，建立最小知识闭环。

范围包括：

- `.rwa` 录入规则
- 来源登记
- 快照存储
- 基础增量判断
- 调用 `graphify` 生成和更新结构化图谱
- 产出最基础的 wiki 沉淀结果
- Obsidian 可读可导航

验收标准：

- 一个本地仓库能被录入、建图并沉淀出基础 wiki
- 一个 PDF / 文档来源能被录入、建图并生成 summary / 主题页
- 同一来源重复同步时，系统能区分未变和已变
- Obsidian 能作为唯一人类入口使用

### 13.2 Phase 2：增量维护闭环

目标是让系统不只是“能导入”，而是“能持续维护”。

范围包括：

- 四层 diff 的正式落盘
- 仓库来源的 `repo -> commit -> file/symbol` 增量链路
- 文档来源的 `document -> snapshot -> section/chunk` 增量链路
- 知识影响传播
- wiki planner 任务化
- lint / audit / 重试 / 局部重建
- 面向 `Codex` / `Trae` 的稳定消费输出

验收标准：

- 仓库提交后，系统能识别新增、修改、删除、重命名，并只更新相关知识
- PDF 大版本更新后，系统能识别大改并标记相关主题待重审
- wiki 更新有明确依据，而不是无差别重写
- 失败来源不会破坏已有知识状态

### 13.3 Phase 3：多模态与主题归档扩展

目标是让知识源类型扩展，但不破坏前两阶段底座。

范围包括：

- 视频 / 音频来源 adapter
- 转写与摘要快照结构
- 自动主题归档
- 更强的领域路由
- 更细的主题知识库组织

该阶段不属于当前实现范围，但必须在一期预留接口。

### 13.4 Phase 4：私有模型消费层

目标是把知识中台真正变成私有智能体的外挂知识底座。

范围包括：

- 私有模型适配器
- 任务上下文裁剪 / 打包
- 领域摘要输出
- 面向模型的问题路由
- 模型回答回流知识库

该阶段不属于当前实现范围，但必须确保前两阶段的数据模型和输出接口不阻断它。

## 15. Workstream 拆分

在 Phase 之下，固定拆成 7 条工作流。Phase 表示时间顺序，Workstream 表示长期稳定边界。

### 14.1 `source-intake`

负责所有知识源进入系统之前的规范化。

包含：

- `.rwa` 目录约束
- 来源登记格式
- 本地文件 / 目录 / 仓库 / 外链录入命令
- 外链副本和指针文件规则
- 来源元数据校验

边界：

- 只解决“来源如何进入系统”
- 不处理图谱和 wiki

### 14.2 `snapshot-state`

负责来源身份、快照、内容指纹和状态落盘。

包含：

- `source_id` 生成规则
- `content_fingerprint` 生成规则
- `source_registry`
- `snapshot_store`
- 状态更新和历史保留策略

边界：

- 只解决“系统知道它是谁、现在是什么版本、历史有哪些版本”

### 14.3 `incremental-diff`

负责四层 diff 和变化分类。

包含：

- `source-level diff`
- `snapshot-level diff`
- `unit-level diff`
- `knowledge-level impact`
- `unchanged / partial_update / major_update / new_source` 判定
- 局部重建与全量重建切换规则

边界：

- 只解决“哪里变了、变化多大、会影响什么”

### 14.4 `graph-foundation`

负责结构化知识底座。

包含：

- 调用 `graphify`
- 抽取器编排与替换接入
- 结构化输出归档
- graph diff 结果管理
- 社区、god node、bridge node、薄弱领域信号抽取
- 图谱结果面向程序和模型的稳定表示

边界：

- 只解决“知识结构长什么样”
- 不负责人类阅读层

### 14.5 `wiki-maintenance`

负责从结构变化到人类知识资产的沉淀。

包含：

- `wiki_planner`
- `wiki_synthesizer`
- 页面类型和模板
- `index.md` 更新
- summary / concept / entity / project / topic 页维护
- 页面依赖映射
- 更新日志和审校入口

边界：

- 只解决“人该看到什么、wiki 如何持续整理”

### 14.6 `agent-consumption`

负责让 `Codex` / `Trae` 在不直接依赖内部实现的情况下消费知识底座。

包含：

- 面向 agent 的稳定输出格式
- `Codex` 消费约定
- `Trae` 消费约定
- 后续私有模型扩展接口占位
- 不同 agent 的最小可用知识视图

边界：

- 只解决“外部 agent 怎么读”
- 不负责底层知识生成

### 14.7 `quality-recovery`

负责系统可靠性。

包含：

- 错误分类
- 失败隔离
- lint / audit
- 重试机制
- 局部重建
- 测试样本
- 验收基线

边界：

- 只解决“系统坏了怎么办、如何验证它没坏”

## 16. Python / Java 边界与扩展设计

一期实现允许优先使用 Python，以利用其在数据处理、文本抽取、图谱编排和快速验证上的优势；但系统长期承载目标必须面向 Java 个人助理项目，以获得更高的可维护性、稳定性和长期演进能力。

因此，本项目从第一阶段开始必须按“Python 可先实现，Java 可后接管”的原则设计，而不能让关键状态、核心流程和 agent 消费接口绑定到 Python 私有实现细节。

### 15.1 设计原则

- 一期允许 `Python-first`
- 一期禁止 `Python-only`
- 所有长期状态和对外消费输出都必须 `language-neutral`
- 二期 Java 个人助理接入应以“消费既有知识底座”为目标，而不是推翻重做

### 15.2 语言无关契约层

以下内容必须定义为稳定 schema 或持久化产物，而不是运行时内部对象：

- `source_registry`
- `snapshot_store`
- `knowledge_state`
- 图谱输出与 graph diff
- wiki 页面依赖映射
- agent-facing 最小消费输出

这些结构必须允许 Python 生成、Java 读取，也允许未来由 Java 重写部分生成逻辑而不破坏既有数据资产。

### 15.3 Python 优先实现层

一期更适合由 Python 承担的模块包括：

- 来源规范化与外链落地
- 快照生成与内容指纹
- 四层 diff 原型
- `graphify` 调用与图谱更新编排
- wiki planner / synthesizer 原型
- lint / audit / 重试等知识维护编排

原因是这些能力与文本处理、脚本编排、文件系统操作和快速迭代强相关，Python 实现成本更低。

### 15.4 Java 后续承载层

后续更适合逐步迁移或接管的模块包括：

- 长期运行的知识服务编排
- 状态存储访问层
- 面向个人助理系统的服务接口
- 稳定的 agent / model 接入层
- 更严格的任务调度、监控和恢复机制

原因是这些能力更接近长期系统承载、服务化和工程稳定性需求，Java 的维护收益更高。

### 15.5 语言边界对 TODO 的影响

后续任务必须补充以下项：

- 定义语言无关 schema
- 定义 Python 产物与 Java 消费之间的文件 / 接口契约
- 明确一期哪些模块允许先用 Python 原型实现
- 明确二期哪些模块优先抽离为 Java 可接管边界

## 19. 结论

`meta_claw` 第一阶段应被设计为一个私有知识中台，而不是一次性图谱生成工具，也不是立即面向所有模型的问答系统。

其核心形态应为：

- `.rwa` 受控录入
- 可编排、可替换的抽取层，当前优先复用 `graphify`
- 面向超大项目和超大文件的分层处理与按需深挖机制
- wiki 面向人的知识沉淀
- Obsidian 作为唯一人类入口
- `Codex` / `Trae` 作为优先消费侧
- 私有模型接入作为后续可扩展能力

核心结论进一步更新为：

- 抽取逻辑必须是框架能力，而不是某个工具实现细节
- 超大来源必须先做结构分层，再做知识深挖，再做展示投影
- 图谱可以保存全结构，但 wiki / Obsidian 只展示人能读的分层视图
- 当前优先让 `Codex` / `Trae` 消费这套知识底座，同时不阻断未来更强抽取器和私有模型接入

这保证系统既能在当前 CLI coding 工作流中立即创造价值，又不会阻断未来向私有智能体长期知识底座演化。
