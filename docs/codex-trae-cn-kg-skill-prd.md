# Codex / Trae CN 知识图谱技能 PRD

## 1. 文档目标

本文定义一个面向 Codex 与 Trae CN 的知识图谱技能产品。

目标不是再做一个新的图谱引擎，而是基于 `.rwa/graphify` 与 `.rwa/llm_wiki`，提供一个可直接调用的技能入口，让用户能以最低门槛完成以下事情：

1. 对任意目录生成知识图谱。
2. 让后续 Agent 优先基于图谱理解项目，而不是盲目搜索原始文件。
3. 在后续阶段逐步把图谱结果接入 LLM Wiki 的知识沉淀流程。

本文重点不在一次性定义所有实现细节，而在于把产品能力拆成多个阶段，确保每个阶段都能独立交付、独立验证、独立创造用户价值。

## 2. 核心价值

这个产品的核心价值不是“功能更多”，而是“让知识结构先于问答发生”。

一句话定义：

> 用户给出一个目录，技能返回一张可导航的知识地图，并让后续 Agent 先按这张地图工作。

它解决的不是传统 RAG 问题，而是三个更基础的问题：

- 用户面对代码仓库、资料目录、知识库项目时，第一步缺少结构化入口。
- Agent 后续问答往往直接 grep 原始文件，效率低且容易失去全局结构。
- 图谱结果常常是一次性产物，没有自然进入持续知识沉淀流程。

因此本产品的核心价值主张应始终保持为三点：

1. 先看结构，再看细节。
2. 图谱不是展示文件，而是后续问答和研究的前置上下文。
3. 图谱结果最终要进入知识生命周期，而不是停留在一次性输出。

## 3. 设计依据

### 3.1 llm-wiki.md 的启发

`llm-wiki.md` 的核心思想不是“把文档喂给 LLM”，而是：

- 原始资料长期积累。
- LLM 将知识持续编译为持久 Wiki。
- Wiki 是知识主体，不是聊天记录。
- 人类负责策展，LLM 负责维护、归档、交叉引用和更新。

因此，本产品不能只停留在“生成一张图”，而要把图谱视为知识沉淀的前置基础设施。

### 3.2 graphify 的现有能力

`.rwa/graphify` 已具备以下成熟能力：

- 多模态文件检测。
- 代码 AST 抽取。
- 文档、图片、多媒体语义抽取。
- 图谱构建、社区检测、报告导出、HTML 可视化。
- 面向 Codex、Trae、Trae CN 的技能安装与平台接入。
- 通过 `AGENTS.md`、技能文件、Hook 机制让图谱结果对后续 Agent 可见。

结论：图谱引擎层应优先复用 `graphify`，不重写。

### 3.3 llm_wiki 的现有能力

`.rwa/llm_wiki` 已具备以下成熟能力：

- `raw/ -> wiki/ -> schema/purpose` 三层结构。
- 两阶段 ingest。
- 基于 `[[wikilink]]` 与 frontmatter 的知识沉淀模型。
- 搜索、图谱、研究、审核、聊天的工作流闭环。
- 本地文件系统与桌面 UI 承载。

结论：知识沉淀层应优先复用 `llm_wiki` 的结构与约束，不重复定义第二套 Wiki 体系。

## 4. 产品定位

本产品应定位为：

> 面向 Codex / Trae CN 的知识图谱技能入口，用于把任意目录或 LLM Wiki 项目转换为可导航、可持续更新、可进入知识生命周期的结构化知识资产。

它不是：

- 一个新的图谱算法项目。
- 一个新的桌面应用。
- 一个替代 `graphify` 或 `llm_wiki` 的全新系统。

它是两者之间的“低门槛启动层”和“场景编排层”。

## 5. 目标用户

### 5.1 用户类型

- 在 Codex 中分析代码仓库和知识目录的开发者。
- 在 Trae CN 中做项目理解、资料梳理和知识沉淀的用户。
- 想从“目录 + 对话”升级到“图谱 + Wiki”工作流的知识管理用户。

### 5.2 核心场景

#### 场景 A：代码仓库理解

用户希望快速看清模块、依赖、社区结构、关键节点和后续问题入口。

#### 场景 B：研究资料理解

用户把论文、网页、图片、Markdown、笔记放在一个目录，希望快速得到概念网络和重点结构。

#### 场景 C：知识库增量更新

用户已经有 LLM Wiki 项目，希望刷新知识图谱，而不必切换多个工具和流程。

#### 场景 D：后续问答和研究前置

用户希望图谱不是静态输出，而是成为后续问答、Review 和 Deep Research 的结构前置条件。

## 6. 设计原则

### 6.1 技能优先

首阶段必须能在 Codex / Trae CN 中独立运行，不依赖桌面 UI。

### 6.2 本地优先

所有输出都落到本地项目目录，便于 Git、Obsidian 和 Agent 重复消费。

### 6.3 复用优先

图谱构建复用 `graphify`，知识结构复用 `llm_wiki`，新产品只负责入口统一、执行编排和阶段接入。

### 6.4 阶段价值清晰

每个阶段都必须回答两个问题：

1. 用户现在就能获得什么价值。
2. 这一阶段完成后，下一阶段是否可以自然接续。

### 6.5 控制复杂度上升方式

不通过删功能控制复杂度，而通过“延迟暴露复杂度”控制复杂度：

- 先做默认路径。
- 再做增量与工程化。
- 再做 LLM Wiki 融合。
- 最后做知识工作流联动。

## 7. 平台约束

### 7.1 Codex

可用机制：

- 技能文件
- `AGENTS.md`
- `.codex/hooks.json`

特点：

- 可通过 Hook 增强“后续问答优先读图谱”的默认行为。
- 若要做并行语义抽取，需要启用 `multi_agent = true`。

### 7.2 Trae CN

可用机制：

- 技能文件
- `AGENTS.md`

特点：

- 无 Hook 兜底。
- 更依赖技能正文与 `AGENTS.md` 规则表达执行行为。

### 7.3 共同基线

两个平台的共同产品基线应是：

- 用户能调用技能。
- 技能能产出标准图谱目录。
- 后续 Agent 能被引导优先消费图谱结果。

Codex 的 Hook 只应作为增强，不应作为核心依赖。

## 8. 总体能力拆分

为了保证产品逐步落地，建议将能力拆为六个阶段。

每个阶段都必须独立可交付、独立可验证，并且在用户视角形成完整闭环。

---

## 9. Phase 1：一键生成知识图谱

### 9.1 阶段目标

建立最低门槛的技能入口。

用户价值：

- 给一个目录，即可生成知识图谱。
- 不需要理解内部脚本和复杂流程。
- 图谱结果能被后续 Agent 直接使用。

### 9.2 范围

- Codex / Trae CN 技能安装。
- 技能触发命令。
- 默认当前目录运行。
- 调用 `graphify` 完成检测、抽取、建图、报告、导出。
- 输出标准 `graphify-out/`。
- 写入 `AGENTS.md`。
- Codex 写入 `.codex/hooks.json`。

### 9.3 输出

至少包括：

- `graphify-out/graph.json`
- `graphify-out/GRAPH_REPORT.md`
- `graphify-out/graph.html`
- `graphify-out/cache/`

### 9.4 用户命令

首版建议只突出默认命令：

```text
$kg .
```

或平台对应触发形式。

### 9.5 验收标准

- 用户执行一次命令即可得到完整图谱输出。
- 输出目录结构稳定。
- 后续 Agent 在同项目中会优先读取 `GRAPH_REPORT.md`。

### 9.6 本阶段不做

- LLM Wiki 项目深度识别。
- 回写 `wiki/`。
- Deep Research 联动。
- Review 联动。
- 复杂模式切换。

---

## 10. Phase 2：增量更新与工程化可用

### 10.1 阶段目标

让技能从“一次性入口”变成“可进入日常工作流的工具”。

用户价值：

- 修改目录后可以低成本刷新图谱。
- 不必重复执行完整流程。
- 目录规模变大后仍然能用。

### 10.2 范围

- `--update`
- 缓存命中提示
- 超大目录检测与降级策略
- 失败恢复与局部继续
- 代码变更后的 AST-only 快速更新
- 可选 `watch` / hook 机制

### 10.3 用户命令

```text
$kg . --update
```

### 10.4 验收标准

- 第二次运行明显快于首次。
- 缓存命中可见。
- 对大目录能给出范围缩小建议或确认提示。
- 单个文件失败不会导致整次运行完全失效。

### 10.5 本阶段不做

- 自动回写 `wiki/log.md`
- 自动写入 `wiki/overview.md`
- 将图谱洞察转成研究页面

---

## 11. Phase 3：LLM Wiki 项目识别与轻融合

### 11.1 阶段目标

让技能开始识别并服务 `llm_wiki` 项目，但不深改桌面端。

用户价值：

- 对已有知识库项目，图谱结果不再是孤立产物。
- 图谱与知识沉淀开始形成连接。

### 11.2 范围

- 自动识别是否为 LLM Wiki 项目。
- 识别规则基于 `wiki/`、`raw/`、`schema.md`、`purpose.md`。
- 图谱构建行为记录到 `wiki/log.md`。
- 为 `wiki/overview.md` 提供图谱摘要更新建议或自动写入。
- 保持 `graphify-out/` 仍为主输出目录。

### 11.3 验收标准

- 对普通目录与 LLM Wiki 项目，技能表现不同但用户不必手工切换系统。
- 图谱结果开始进入知识库生命周期。
- 不要求修改 `llm_wiki` UI 即可完成使用。

### 11.4 本阶段不做

- 直接进入 `llm_wiki` 桌面应用内部状态管理
- Chat / Search / Research 深层联动
- Review Item 自动生成

---

## 12. Phase 4：图谱洞察转知识动作

### 12.1 阶段目标

让图谱从“展示结构”升级为“推动知识增长的动作入口”。

用户价值：

- 不只是看到图谱，而是知道下一步研究什么、补什么、连什么。

### 12.2 范围

- 从图谱中提取知识空白、桥接节点、高价值连接。
- 生成下一步研究建议。
- 可选生成 `wiki/queries/graph-insights-<date>.md`
- 为后续 Deep Research 提供输入主题。

### 12.3 验收标准

- 用户运行后不仅看到图，还能得到下一步知识动作建议。
- 图谱开始真正成为知识生产前置。

### 12.4 本阶段不做

- 自动完成完整 Deep Research 流程
- 与桌面端所有视图深度联动

---

## 13. Phase 5：与 llm_wiki 工作流联动

### 13.1 阶段目标

把图谱结果接入 `llm_wiki` 的 ingest / query / research 工作流。

用户价值：

- 图谱不只是“构建结果”，而是问答和研究前的结构上下文。

### 13.2 范围

- query 前优先消费图谱摘要与结构结果。
- research 结果回写后触发图谱刷新。
- 图谱辅助 Review / Research 建议。
- 图谱结果进入查询上下文组装过程。

### 13.3 验收标准

- 图谱成为 LLM Wiki 工作流的一部分，而非并列工具。
- 后续查询与研究质量因结构上下文而明显改善。

### 13.4 本阶段不做

- 团队协同
- 云端托管
- 全新 UI 系统重构

---

## 14. Phase 6：平台完善与高级能力

### 14.1 阶段目标

补齐跨平台、扩展性和高级能力。

用户价值：

- 更稳、更强、更适合长期使用。

### 14.2 范围

- Codex / Trae CN 平台差异进一步细化。
- MCP / 查询接口增强。
- 更强多模态与高级导出。
- 更深层桌面端或团队能力扩展。

### 14.3 验收标准

- 平台差异被封装到安装与能力层，而非暴露给普通用户。
- 产品具备长期扩展空间。

### Phase 4 对应功能

- 研究建议

## 16. 功能需求总表

### 16.1 技能安装

系统应支持：

- 安装 Codex 技能版本
- 安装 Trae CN 技能版本
- 写入平台技能目录
- 写入项目级 `AGENTS.md`
- 对 Codex 写入 `.codex/hooks.json`

### 16.2 技能触发

系统应支持：

- 默认当前目录
- 指定路径
- 后续支持 `--update`

### 16.3 输入检测

系统应支持：

- 统计文件类型和规模
- 检测敏感路径和应跳过目录
- 大目录提示缩小范围
- 音视频先转录再进入后续抽取

### 16.4 图谱构建

系统应支持：

- 代码 AST 抽取
- 非代码语义抽取
- 图谱构建
- 社区检测
- 报告与导出

### 16.5 增量能力

系统应支持：

- 缓存命中
- update
- 快速重建

### 16.6 LLM Wiki 对接

系统应支持：

- 项目识别
- 轻量回写 `wiki/log.md`
- 图谱摘要进入 `wiki/overview.md`

### 16.7 知识动作

系统应支持：

- 图谱洞察
- 研究建议
- 后续知识动作入口

### 16.8 工作流联动

系统应支持：

- 图谱作为后续问答前置
- 图谱进入查询与研究链路

## 17. 非功能需求

### 17.1 可靠性

- 缺少依赖时有明确提示
- 局部失败允许继续
- 不破坏原始资料

### 17.2 可解释性

- 清晰区分 EXTRACTED / INFERRED / AMBIGUOUS
- 保留置信度与来源

### 17.3 可维护性

- 技能说明短而稳定
- 平台差异尽量收敛在安装层

### 17.4 兼容性

- 与现有 `graphify` 输出兼容
- 与现有 `llm_wiki` 结构兼容
- 与 Git / Obsidian 工作流兼容

### 17.5 性能

- 优先走增量路径
- 大目录具备缩小范围策略
- 对平台并行能力有显式配置

## 18. 技术架构建议

建议分四层：

1. 技能层
2. 执行编排层
3. 图谱引擎层
4. 知识库融合层

### 18.1 技能层

负责：

- Codex / Trae CN 技能文案
- 参数入口
- 平台安装
- 常驻规则写入

建议复用：

- `.rwa/graphify/graphify/skill-codex.md`
- `.rwa/graphify/graphify/skill-trae.md`
- `.rwa/graphify/graphify/__main__.py`

### 18.2 执行编排层

负责：

- 检测输入路径
- 判断当前阶段能力入口
- 分派 AST 与语义抽取
- 汇总输出
- 增量与 watch 逻辑

### 18.3 图谱引擎层

建议直接复用 `graphify`：

- `detect.py`
- `extract.py`
- `build.py`
- `cluster.py`
- `analyze.py`
- `report.py`
- `export.py`

### 18.4 知识库融合层

负责：

- LLM Wiki 项目识别
- 图谱结果向 `wiki/` 资产映射
- 为后续 query / research / review 提供结构入口

## 19. 风险与控制策略

### 19.1 边界混乱

风险：

- 做成半个 `graphify`
- 又做成半个 `llm_wiki`

策略：

- 图谱构建复用 `graphify`
- Wiki 生命周期复用 `llm_wiki`
- 新产品只负责入口统一、阶段编排和轻量融合

### 19.2 平台机制不一致

风险：

- Codex 有 Hook，Trae CN 没有

策略：

- 共同基线建立在技能 + `AGENTS.md`
- Hook 只做增强

### 19.3 大语料性能与成本

风险：

- 多模态和大规模目录代价过高

策略：

- 强制检测摘要
- 默认支持增量
- 支持按子目录运行

### 19.4 过早深度接入 llm_wiki

风险：

- 首版就改 UI 和状态模型，范围急剧膨胀

策略：

- 先做文件级和规则级对接
- 桌面端联动后置到后续阶段

## 20. 成功标准

### 20.1 Phase 1 成功标准

- 用户能在 Codex / Trae CN 中一条命令生成知识图谱
- 后续 Agent 会优先读取图谱结果

### 20.2 Phase 2 成功标准

- 用户愿意重复使用 `--update`
- 日常变更后刷新图谱成本明显降低

### 20.3 Phase 3 成功标准

- 在 LLM Wiki 项目上运行后，图谱结果进入知识库生命周期

### 20.4 Phase 4 成功标准

- 用户从图谱中直接获得下一步研究或补全建议

### 20.5 Phase 5 成功标准

- 图谱成为查询和研究前置上下文的一部分

### 20.6 总体成功标准

- 产品成为从“目录”到“知识系统”的第一步
- 用户面对新项目时先运行图谱技能，再进入问答或沉淀流程

## 21. 结论

这项产品的正确方向不是一次性做全，而是按阶段推进：

- 第一阶段解决“有没有入口”
- 第二阶段解决“能不能进入日常工作流”
- 第三阶段解决“能不能接到知识库”
- 第四和第五阶段解决“能不能真正驱动知识增长”

因此，产品主线应始终保持不变：

> `graphify` 负责“看见结构”，`llm_wiki` 负责“沉淀知识”，而这个技能产品负责“让用户在 Codex / Trae CN 中随手启动这条结构化知识链路，并按阶段逐步增强它”。
