# Graphify + LLM Wiki 需求文档

## 1. 文档目的

本文基于对 `.rwa/graphify` 与 `.rwa/llm_wiki` 的代码、README 与核心模块的分析，提炼两者的共同产品方向，形成一份可用于后续设计、实现与排期的需求文档。

目标不是重复源码细节，而是回答四个问题：

1. 这两个项目分别解决什么问题。
2. 两者组合后应当形成怎样的产品形态。
3. 核心用户场景与必须能力是什么。
4. 后续实现应优先满足哪些功能与约束。

## 2. 现状提炼

### 2.1 graphify 的本质

`graphify` 是一个“知识结构提取引擎”，核心价值是把零散语料转换为可查询、可解释、可复用的知识图谱输出。

当前已具备的关键能力：

- 支持代码、Markdown、PDF、图片、音视频等多模态输入。
- 对代码优先走 AST/静态分析，减少 LLM 依赖与成本。
- 对文档、多媒体内容补充语义抽取，形成实体、关系、推断与置信度标记。
- 将结果构建为 NetworkX 图，并执行社区发现、分析、报告生成与 HTML/JSON 导出。
- 输出 `graphify-out/graph.json`、`GRAPH_REPORT.md`、`graph.html` 等持久化结果。
- 提供增量更新、缓存、watch、MCP server、安装到不同 AI coding 平台的能力。

一句话总结：`graphify` 负责“把原始资料变成结构地图”。

### 2.2 llm_wiki 的本质

`llm_wiki` 是一个“持续积累型知识工作台”，核心价值是让 LLM 不再每次临时检索原始文档，而是持续把知识编译进 Wiki，再基于 Wiki 回答问题、补齐空白、驱动研究。

当前已具备的关键能力：

- Tauri 桌面应用，支持项目创建、打开、切换与本地文件系统管理。
- 以 `raw sources -> wiki -> schema/purpose` 的三层结构维护知识库。
- 两阶段 ingest：先分析，再生成 wiki 页面。
- 增量缓存、持久化队列、失败恢复、活动面板、自动保存。
- Token 搜索、图相关性扩展、可选向量检索、上下文预算控制。
- Wiki 图谱构建、Louvain 社区发现、图谱可视化与图谱洞察。
- Deep Research、Review Queue、Web Clipper、Chat 多会话、引用回写 Wiki。
- 兼容 Obsidian 工作流，Wiki 目录本身可作为可读可编辑资产。

一句话总结：`llm_wiki` 负责“把结构化知识持续沉淀为可维护 Wiki，并面向查询、研究和编辑消费这些知识”。

### 2.3 两者的关系

两者并不是重复建设，而是上下游关系：

- `graphify` 偏底层，负责知识抽取、图谱构建、结构分析、结构导航。
- `llm_wiki` 偏上层，负责知识沉淀、任务工作流、交互界面、研究闭环。

最合理的统一产品定位应为：

> 一个以知识图谱为骨架、以持久 Wiki 为主体、以 LLM 为自动维护引擎的个人/团队知识操作系统。

## 3. 产品定位

### 3.1 产品名称建议

暂定：知识图谱驱动的 LLM Wiki 平台。

### 3.2 产品愿景

让用户把“原始资料集合”逐步演化为“可解释、可追踪、可持续增长的知识资产”，避免每次问答都重新做一次临时 RAG。

### 3.3 核心价值主张

- 从“临时检索”升级到“持续编译知识”。
- 从“文件堆积”升级到“结构化知识网络”。
- 从“单次问答”升级到“持续积累、持续修正、持续研究”。
- 从“LLM 黑盒生成”升级到“有来源、有链接、有图谱依据”的知识维护。

## 4. 目标用户

### 4.1 核心用户

- 长周期研究者：论文、报告、网页、笔记持续汇总的人。
- 高密度阅读者：读书、课程、专题学习中需要持续沉淀认知的人。
- 技术团队/产品团队：需要将项目文档、会议记录、方案、源码知识持续汇聚的人。
- 个人知识管理重度用户：已经使用 Obsidian、Markdown、目录结构管理知识的人。

### 4.2 用户特征

- 愿意维护自己的资料来源，但不愿手工维护大量 Wiki 页面。
- 希望知识资产长期存在，而非散落在聊天记录里。
- 需要解释性、来源追踪、跨文档关联，而不只是单轮回答。

## 5. 核心问题定义

当前主流方案存在以下问题：

- RAG 每次都从原始资料重新检索，知识不会自然累积。
- 资料越多，人工整理成本越高，Wiki 越容易失效。
- 普通笔记系统有存储能力，但缺少自动结构化、自动关联与自动维护能力。
- 单纯图谱系统能看结构，但不能承担完整的知识生产与查询工作流。

因此产品必须同时解决三个问题：

1. 如何把原始资料自动转成结构化知识。
2. 如何让结构化知识持续沉淀成可读、可编辑、可复用的 Wiki。
3. 如何围绕 Wiki 建立查询、研究、审阅、补全的闭环。

## 6. 产品范围

### 6.1 In Scope

- 多源资料导入。
- 自动抽取实体、概念、关系与上下文结构。
- 生成知识图谱、社区结构、报告与可视化。
- 自动生成与更新 Wiki 页面。
- 基于 Wiki 和图谱的搜索、问答、研究与审阅。
- 资料来源追踪、缓存、增量更新、持久化状态。
- 本地优先的桌面工作流与文件夹项目管理。

### 6.2 Out of Scope

- 云端多人协同编辑平台。
- 完整在线 SaaS 托管。
- 通用企业级权限系统。
- 与外部 IM/CRM/工单系统的深度双向集成。

## 7. 业务流程

### 7.1 主流程

1. 用户创建或打开一个知识项目。
2. 用户导入文件、网页剪藏、文件夹或研究结果。
3. 系统解析原始资料并做增量检查。
4. 系统执行知识抽取，构建实体、关系与图谱。
5. 系统基于抽取结果生成或更新 Wiki 页面、索引、概览、日志。
6. 用户通过搜索、聊天、图谱、审阅、研究面板消费与修正知识。
7. 新产生的研究结果或高价值问答再次回流到 Wiki，形成闭环。

### 7.2 支撑流程

- 图谱更新：代码或资料变化后支持增量重建。
- 审阅流程：LLM 标注不确定项，由用户异步处理。
- 深度研究：从知识空白或桥接点出发发起网络研究，并自动回流。
- 向量增强：可选开启 embedding，提高语义检索召回。

## 8. 功能需求

### 8.1 项目与知识库管理

系统应支持：

- 创建知识项目并生成基础目录结构。
- 打开已有项目并校验必要目录与配置文件。
- 持久化最近项目、当前项目、语言与模型配置。
- 在桌面端浏览项目文件树与 Wiki 树。

验收标准：

- 用户关闭应用后再次打开，能够恢复上次项目与关键配置。
- 项目打开后可以立即浏览 `wiki/` 与 `raw/` 内容。

### 8.2 资料导入与采集

系统应支持：

- 导入单文件、整目录、网页剪藏结果。
- 保留目录上下文，作为分类与生成提示的一部分。
- 支持 Markdown、文本、代码、图片、PDF、音视频等资料类型。
- 对接本地 clip server，自动发现剪藏内容并触发后续流程。

验收标准：

- 新资料进入项目后，用户能看到导入状态、目标路径与处理结果。
- 支持对同一资料重复导入时做去重或增量跳过。

### 8.3 知识抽取与图谱构建

系统应支持：

- 对代码资料执行 AST/静态结构抽取。
- 对文档、多媒体资料执行语义抽取。
- 生成节点、边、关系类型、来源位置、置信度标签。
- 进行社区发现、枢纽节点识别、桥接关系识别与图谱分析。
- 导出可视图谱、查询图谱与结构分析报告。

验收标准：

- 每个图谱节点和边都应能追溯到来源文件或推断依据。
- 报告中至少包含核心节点、社区结构、值得关注的连接与建议问题。

### 8.4 Wiki 自动生成与维护

系统应支持：

- 对新资料执行“两阶段 ingest”。
- 基于分析结果生成 source/entity/concept/synthesis/query 等页面。
- 自动更新 `index.md`、`overview.md`、`log.md`。
- 为页面写入 frontmatter，包括 `type`、`title`、`sources` 等字段。
- 当 LLM 漏写关键页面时执行兜底生成。

验收标准：

- 处理一份新资料后，Wiki 中至少新增来源页，且索引与日志得到更新。
- 生成页面之间应尽可能通过 `[[wikilink]]` 建立连接。

### 8.5 增量处理与缓存

系统应支持：

- 基于文件内容哈希跳过未变化资料。
- 持久化 ingest queue，支持恢复、取消、重试。
- 对图谱生成过程保留缓存，避免重复抽取未变化内容。

验收标准：

- 应用重启后未完成任务可继续恢复。
- 未变化资料再次处理时应显著快于首次处理。

### 8.6 搜索与检索增强

系统应支持：

- 基于标题与内容的 token 搜索。
- 支持中英文混合查询与中文分词近似策略。
- 基于知识图谱做相关节点扩展。
- 可选使用向量检索提升召回。
- 根据上下文预算拼装问答上下文。

验收标准：

- 用户输入问题后，系统能返回排序后的页面候选，并可用于后续问答。
- 在禁用 embedding 时仍可完成可用检索链路。

### 8.7 Chat 问答与知识回写

系统应支持：

- 多会话聊天。
- 为回答附带引用来源。
- 控制历史消息深度。
- 将高价值回答保存为 Wiki 页面并重新进入 ingest 流程。

验收标准：

- 用户切换会话后历史消息保持独立。
- 保存后的问答内容能够在后续图谱与 Wiki 中被再次利用。

### 8.8 图谱可视化与图谱洞察

系统应支持：

- 展示 Wiki 图谱。
- 支持按页面类型或社区着色。
- 提供节点大小、边强度、邻居高亮、缩放与适配视图。
- 输出 surprising connections、isolated pages、sparse communities、bridge nodes 等洞察。

验收标准：

- 用户点击洞察项后，图谱中对应节点与边应被高亮。
- 社区信息应至少包含规模、代表节点与凝聚度。

### 8.9 Deep Research

系统应支持：

- 从用户主题或图谱洞察触发研究任务。
- 生成或接收多条搜索查询。
- 汇总搜索结果并由 LLM 合成为研究页。
- 将研究结果写入 `wiki/queries/` 并自动回流到 Wiki 主体。

验收标准：

- 研究任务应具备状态流转：排队、搜索、综合、保存、完成、失败。
- 研究结果页必须包含引用来源。

### 8.10 Review 人审机制

系统应支持：

- 在 ingest 阶段生成待审事项。
- 限定待审动作类型，如创建页面、深度研究、跳过。
- 将待审问题与资料来源关联。
- 用户异步完成审阅，不阻塞主 ingest 流程。

验收标准：

- 审阅项必须可持久化。
- 审阅项应可区分已处理与未处理状态。

### 8.11 模型与提供方配置

系统应支持：

- 配置 OpenAI、Anthropic、Google、Ollama、兼容 OpenAI 的自定义服务。
- 配置 Web Search provider 与 embedding provider。
- 根据项目保存模型参数与语言设置。

验收标准：

- 切换配置后，无需重新创建项目即可生效。
- 本地模型与远程模型均可走同一工作流。

## 9. 非功能需求

### 9.1 可解释性

- 所有知识生成结果应尽可能保留来源、链接与置信度。
- 系统必须明确区分“直接抽取”和“合理推断”。

### 9.2 可持续积累

- 所有核心资产应以本地文件形式持久化，而非只存在内存或聊天记录中。
- 生成的 Wiki 与图谱输出应可被 Git、Obsidian 与外部工具复用。

### 9.3 性能

- 大部分未变更内容应支持跳过处理。
- UI 不应因长时间 ingest 或研究任务完全阻塞。
- 图谱与搜索结果应在可接受时间内可视化和返回。

### 9.4 本地优先与安全

- 原始资料默认保存在本地项目目录。
- 文件操作应尽量限制在项目范围内。
- 本地向量库、缓存、日志不得破坏原始资料层。

### 9.5 可扩展性

- 抽取器、模型提供方、搜索提供方、向量后端应可替换。
- 图谱既可作为 UI 视图，也可作为外部 MCP/查询接口使用。

## 10. 信息架构建议

统一后的产品建议分为五层：

1. 原始资料层：`raw/`
2. 知识抽取层：graphify pipeline / ingest pipeline
3. 结构资产层：`graph.json`、`GRAPH_REPORT.md`、community、relevance graph
4. 知识内容层：`wiki/`、`index.md`、`overview.md`、`log.md`
5. 交互工作台层：Search、Chat、Graph、Review、Research、Settings

其中：

- `graphify` 更适合作为第 2、3 层的通用引擎。
- `llm_wiki` 更适合作为第 4、5 层的产品外壳。

## 11. 版本优先级建议

### P0

- 项目管理
- 资料导入
- 两阶段 ingest
- 基础 Wiki 生成
- 增量缓存
- 基础搜索
- 多会话 Chat
- 基础图谱展示

### P1

- 图谱洞察
- Deep Research
- Review Queue
- 向量检索
- Web Clipper 自动回流

### P2

- 更强的多模态抽取
- 团队协作能力
- 更细粒度的工作流编排
- 外部 MCP / API 化能力增强

## 12. 成功指标建议

- 单个项目中，新增资料后的自动生成成功率。
- 被缓存跳过的资料占比。
- 用户查询时命中的有效页面数与引用覆盖率。
- 图谱中孤立页面占比是否持续下降。
- Deep Research 结果回流后新增有效连接数。
- 用户保留并持续使用同一知识库的周期长度。

## 13. 风险与注意事项

- 过度依赖 LLM 生成可能导致结构漂移，需要 schema/purpose 持续约束。
- Wiki 页面过多后，`index.md` 与全量读取策略可能出现规模瓶颈。
- 图谱与 Wiki 若分别演化，可能出现结构不一致，需要统一更新机制。
- Deep Research 引入外部搜索后，来源质量与去重策略会影响结果可信度。
- 本地桌面应用跨平台能力强，但后续团队协作和同步能力不足。

## 14. 结论

`.rwa/graphify` 与 `.rwa/llm_wiki` 的组合，不应被理解为“一个图谱工具 + 一个笔记应用”，而应被定义为：

> 一个以原始资料为输入、以知识图谱为结构骨架、以持久 Wiki 为知识主体、以 LLM 为自动维护引擎的知识生产系统。

对后续产品设计来说，最关键的不是继续堆功能，而是保证以下主线始终成立：

- 新资料能稳定进入系统。
- 系统能把资料转成结构化知识。
- 结构化知识能稳定沉淀进 Wiki。
- 用户能通过搜索、问答、图谱、研究和审阅持续消费并反哺知识库。

只要这条闭环成立，产品就具备长期复利。
