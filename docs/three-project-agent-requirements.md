# 三项目对比后的 Agent 平台需求文档

## 1. 文档目标

本文档仅做需求分析，不做技术设计，不限定具体实现细节。

分析对象：

- `.rwa/claudecode/src/tools`
- `.rwa/CowAgent`
- `.rwa/expert_project`

目标：

- 分析 `claudecode` 的核心实现与 Agent harness 理念如何落地
- 对比 `CowAgent` 与 `expert_project`
- 提炼三个项目各自最值得继承的优点
- 生成一份统一的产品需求文档

## 2. 三个项目的核心定位

### 2.1 claudecode

`claudecode` 的核心不是“功能堆砌”，而是一个偏底层的 Agent harness。

从 `src/tools` 子图可见，它最强的社区集中在：

- Agent Delegation
- Tool UI Rendering
- Agent Runtime Core
- Bash Safety Parsing
- PowerShell Permissions
- Bash Permissions
- Path Validation
- Shell Execution Semantics
- Web Fetch Controls
- Skill Execution
- Multi Agent Spawn
- Agent Memory Snapshot

关键特征：

- 强调工具运行的安全边界
- 强调工具并发与串行调度
- 强调 subagent / teammate / worktree / remote 等隔离与委派模型
- 强调权限模式、审计、可恢复执行和进度反馈
- 强调把“Agent 能做什么”变成一个可控的运行时框架

它更像：

**一个面向编码代理的执行控制内核。**

### 2.2 CowAgent

`CowAgent` 更偏完整产品形态。

它的优势是：

- 多通道接入成熟
- Prompt、技能、记忆、知识、CLI、插件都已经形成闭环
- 更贴近真实运营和长期运行的助理平台

它更像：

**一个已经具备产品化广度的超级助理平台。**

### 2.3 expert_project

`expert_project` 更偏清晰抽象和模块边界。

它的优势是：

- Runtime、Gateway、Tool、Knowledge、Memory 职责清晰
- 抽象干净，适合企业级重构和平台化演进
- 模块命名、接口边界、分层思路更稳定

它更像：

**一个适合沉淀为企业级平台骨架的架构原型。**

## 3. claudecode 的核心实现与 Agent Harness 落地

这里把 “herness” 统一理解为 **harness**，即 Agent 的执行控制框架。

### 3.1 harness 的本质

`claudecode` 的 harness 并不是单个类，而是一组围绕工具与子代理的控制机制。

它关注的问题是：

- 哪些工具可以执行
- 什么情况下可以并发
- 哪些操作必须串行
- 哪些命令是危险的
- 子代理如何隔离
- 子代理如何继承上下文和权限
- 结果如何回流到主代理
- 用户如何看到执行进度

### 3.2 harness 在代码中的具体落地点

#### 3.2.1 AgentTool 作为主入口

`AgentTool.tsx` 是子代理启动和委派的核心入口。

它负责：

- 统一接收 “启动 agent” 请求
- 选择同步、后台、teammate、worktree、remote 等不同执行模式
- 注入权限模式、模型、cwd、team_name、isolation 等约束
- 把子代理执行纳入统一生命周期管理

这意味着 `AgentTool` 不是普通工具，而是 **Agent 的调度入口**。

#### 3.2.2 runAgent 作为运行时装配层

`runAgent.ts` 负责真正把一个 agent 跑起来。

它落地了：

- 工具池解析
- MCP server 初始化
- system prompt 组装
- hooks 注册
- transcript 与 metadata 存储
- agent 上下文创建

这一步使 harness 从“调用入口”进入“运行时装配”。

#### 3.2.3 toolOrchestration / StreamingToolExecutor 作为执行调度层

`toolOrchestration.ts` 和 `StreamingToolExecutor.ts` 体现了 harness 的执行控制思想。

落地方式：

- 把工具按 concurrency-safe 与非 concurrency-safe 分组
- 允许只读/安全工具并发
- 对有副作用工具进行串行化
- 保证消息结果按原始顺序回放
- 在某个工具报错时中断并取消其他兄弟工具

这说明 `claudecode` 不是“能调用工具”就算完成，而是把 **工具调度本身产品化** 了。

#### 3.2.4 Bash / PowerShell 安全链路

在 `bashPermissions.ts`、`pathValidation.ts`、`powershellPermissions.ts`、`powershellSecurity.ts` 中，harness 把命令执行安全显式建模。

包括：

- 权限模式校验
- 只读约束
- 路径约束
- git 内部路径保护
- 危险命令检查
- 编码命令、下载执行链、动态命令名检查

这部分说明它的理念是：

**先控制风险，再开放能力。**

#### 3.2.5 spawnMultiAgent 作为多代理协作层

`shared/spawnMultiAgent.ts` 将 harness 推进到多代理协作。

它落地了：

- teammate 命名与身份
- 会话继承
- CLI 参数继承
- model 继承
- tmux / in-process / pane 等多种协作后端
- 团队上下文与 swarm 运行方式

这说明 `claudecode` 的多代理不是临时加的，而是运行时一等能力。

#### 3.2.6 Agent Memory Snapshot 作为上下文延续层

`agentMemory.ts` 与 `agentMemorySnapshot.ts` 体现了 harness 的另一层落地：

- 子代理不是完全无状态
- 需要在多轮、多代理、远程或 fork 场景中保持可恢复上下文
- memory snapshot 成为跨执行边界的状态承载方式

### 3.3 对产品的启发

`claudecode` 带来的最大启发不是“再加几个工具”，而是：

- Agent 平台必须有完整的执行控制层
- 工具系统必须是受控系统，而不是裸调用集合
- 多代理、隔离、权限、审计、回放、进度，本身就是核心产品能力

## 4. 三个项目的优点提炼

### 4.1 claudecode 的优点

- Agent harness 最强，执行控制最完整
- 工具安全边界建模最成熟
- 并发/串行调度机制清晰
- 多代理协作与隔离策略成熟
- 进度、结果、UI 渲染与运行状态整合较好
- 适合做高可靠 Agent Runtime 内核参考

### 4.2 CowAgent 的优点

- 产品能力覆盖最完整
- 多通道接入最成熟
- 技能、记忆、知识、PromptBuilder、CLI 已形成完整闭环
- 更接近真实用户和运营场景
- 适合作为需求范围和产品形态参考

### 4.3 expert_project 的优点

- 抽象最清晰
- Runtime、Gateway、Tool、Memory、Knowledge 边界最好
- 更容易演化为稳定的平台骨架
- 更适合做企业级架构治理
- 适合作为系统骨架和模块拆分参考

## 5. 统一产品结论

目标产品不应复制任何一个项目，而应融合三者优点：

- 用 `expert_project` 提供平台骨架
- 用 `CowAgent` 提供产品能力广度
- 用 `claudecode` 提供 Agent harness 与运行控制深度

一句话总结：

**骨架学 expert_project，产品面学 CowAgent，执行控制学 claudecode。**

## 6. 产品目标

构建一个可长期运行、可扩展、可控、可审计的 Agent 平台。

产品目标包括：

- 支持多渠道接入
- 支持多模型与多代理协作
- 支持工具、技能、记忆、知识的一体化运行
- 支持高风险能力的权限控制与审批
- 支持面向企业场景的运维、审计和可观测

## 7. 目标用户

- 企业内部 AI 平台团队
- 需要私有化 Agent 的业务团队
- 需要多渠道智能助理的运营团队
- 需要构建编码代理、业务代理、流程代理的开发团队

## 8. 核心业务场景

### 8.1 多渠道助手场景

- 用户从 Web、企业微信、飞书、钉钉、终端等入口发起请求
- 系统统一处理消息协议
- Agent 输出文本、文件、卡片、图片等多种结果

### 8.2 复杂任务执行场景

- 用户输入复杂目标而非单步指令
- Agent 自主拆解步骤
- Agent 调用工具逐步执行
- 必要时拉起子代理协作

### 8.3 长期记忆与知识增强场景

- Agent 记住用户偏好、历史上下文和关键事实
- Agent 能从知识库中检索资料并注入决策过程
- Agent 对长期任务具有连续性

### 8.4 企业安全治理场景

- 高风险工具调用需要审批
- 敏感命令执行需要策略校验
- 所有关键操作需要审计留痕

### 8.5 多代理协作场景

- 主代理负责统筹
- 子代理负责局部任务
- 支持后台运行、隔离执行、结果回流与汇总

## 9. 需求范围

### 9.1 接入层需求

- 支持 Web Chat
- 支持企业微信、飞书、钉钉、终端
- 支持统一消息协议
- 支持通道独立启停与配置
- 支持多通道并发运行

### 9.2 Agent Runtime 需求

- 支持普通对话模式
- 支持多步 Agent 模式
- 支持工具调用循环
- 支持流式输出
- 支持中断、超时、重试、取消
- 支持上下文裁剪与上下文恢复

### 9.3 Agent Harness 需求

- 工具必须区分并发安全与非并发安全
- 支持工具批量并发执行
- 支持危险工具串行执行
- 支持工具状态跟踪：排队、执行中、完成、取消、失败
- 支持权限模式：只读、可编辑、自动审批、强审批等
- 支持高风险命令与路径校验
- 支持执行进度回放和结果有序回流

### 9.4 工具系统需求

- 支持内置工具与扩展工具
- 支持工具注册、启停、配置和权限控制
- 支持文件、Shell、PowerShell、HTTP、浏览器、知识、记忆、定时任务等工具
- 支持工具结果标准化输出
- 支持工具调用审计

### 9.5 子代理与多代理需求

- 支持子代理同步执行
- 支持后台代理执行
- 支持隔离执行
- 支持工作区隔离或等价隔离机制
- 支持团队/角色化代理
- 支持代理结果汇总、转交与中途接管

### 9.6 技能系统需求

- 支持技能加载、安装、启用、禁用、升级
- 支持技能元数据与依赖校验
- 支持技能与工具、知识、记忆联动
- 支持技能市场或仓库来源
- 支持技能按租户/角色授权

### 9.7 记忆系统需求

- 支持会话记忆
- 支持长期记忆
- 支持摘要压缩
- 支持关键词检索和向量检索
- 支持不同作用域的记忆隔离
- 支持跨代理的记忆快照或上下文继承

### 9.8 知识库需求

- 支持文档导入、分块、索引、检索
- 支持文档元数据
- 支持知识结果回注到 Agent 上下文
- 支持知识图谱或引用关系视图

### 9.9 会话与任务管理需求

- 支持多会话管理
- 支持会话标题生成
- 支持会话恢复
- 支持任务状态追踪
- 支持后台任务列表与结果查看

### 9.10 管理后台需求

- 管理渠道
- 管理模型
- 管理工具
- 管理技能
- 管理知识库
- 查看会话和任务
- 查看审计日志与成本数据

### 9.11 安全与合规需求

- 高风险工具审批
- 命令安全校验
- 文件路径保护
- 凭证脱敏
- 操作审计
- 角色权限控制

### 9.12 可观测需求

- 记录模型调用耗时与 token 消耗
- 记录工具调用链
- 记录子代理执行链
- 记录错误与中断原因
- 提供任务进度与执行状态可视化

## 10. 非功能需求

- 可私有化部署
- 可横向扩展
- 模块可插拔
- 对外接口稳定
- 关键链路高可观测
- 关键执行路径可审计
- 高风险操作默认保守

## 11. 产品优先级建议

### P0

- 多渠道消息入口
- Agent Runtime
- Tool Harness
- 基础工具系统
- 会话与任务管理
- 安全校验与审批

### P1

- 技能系统
- 长期记忆
- 知识库
- 多代理协作
- 管理后台

### P2

- 技能市场
- 图谱化知识浏览
- 高级团队协作
- 更细粒度的组织与租户能力

## 12. 最终结论

三个项目分别代表三个不同方向的最优样本：

- `claudecode`：最强执行控制与 Agent harness
- `CowAgent`：最强产品完整度与平台能力广度
- `expert_project`：最强抽象清晰度与系统骨架

因此目标产品的需求定义应是：

**一个具备多渠道、多代理、强工具控制、长期记忆、知识增强、技能扩展和企业级治理能力的 Agent 平台。**

它既不是单纯聊天机器人，也不是单纯编码代理，而是：

**可运行、可协作、可审计、可扩展、可长期演进的统一智能体平台。**
